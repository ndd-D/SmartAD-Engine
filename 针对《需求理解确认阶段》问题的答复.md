## 针对《需求理解确认阶段》问题的答复

以下是根据你的项目文档和常规技术实践给出的答复，你可以直接使用：

---

### 一、技术架构层面

#### 1. ClickHouse 与 MySQL 的数据同步
**答复**：采用**双写+定时任务补偿**机制。
- 用户行为日志实时写入 ClickHouse（通过 `/api/data/log/report`）
- 同时写入 MySQL 作为备份和关联查询
- 每日凌晨 2 点通过定时任务对账，确保数据一致性
- **第一阶段**：先实现 ClickHouse 单写，MySQL 备份后续再加

#### 2. Redis 缓存设计
**答复**：第一阶段**先不实现**，用本地缓存（Caffeine/Guava）替代。
- 缓存键设计示例：`smartad:crowd:{crowdTag}`、`smartad:history:{crowdTag}:{channel}`
- 过期策略：24 小时主动过期
- 更新机制：写入时删除缓存，下次查询重新加载
- **第二阶段**再引入 Redis 实现分布式缓存

#### 3. ai-token 鉴权机制
**答复**：第一阶段用**固定静态 token**（就是文档里的 `sk_ai_ad_202603`）。
- Java 后端硬编码校验：`"sk_ai_ad_202603".equals(aiToken)`
- **后续阶段**可升级为 JWT 动态生成+定期刷新

---

### 二、接口层面

#### 4. 渠道API对接
**答复**：第一阶段**全部模拟实现**，返回固定成功响应。
- 例如创建广告接口直接返回：`{"code":200,"data":{"adId":"AD_MOCK_001","status":"投放中"}}`
- **第三阶段**再对接真实渠道 API（抖音/微信/头条）

#### 5. AI策略生成的数据依赖
**答复**：**需要你先初始化模拟数据**。
- 在 MySQL 中预置 5-10 条人群画像数据
- 预置 10-20 条历史投放效果数据
- 让 AI 能真实调用并拿到数据做决策

#### 6. 前端状态轮询机制
**答复**：第一阶段用**轮询**，间隔 **3秒**。
- 轮询接口：`/api/strategy/command/status?commandId=xxx`
- 3 秒是比较平衡的间隔，不会给后端太大压力，用户体验也尚可
- **后续阶段**可考虑 WebSocket 做实时推送

### 三、AI层面

#### 7. DeepSeek API Key
**答复**：**该 Key 仅为示例，不可用**。
- **Key 是真实的，可以直接使用**：`sk-e88cf58b6b0648b287f782331cae6026`
- 地址：https://api.deepseek.com/v1

#### 8. 模型分级调用的实现

**答复**：统一使用 **`deepseek-chat`** 模型。
- 经核实，DeepSeek 官方确实没有 `deepseek-chat-lite`
- 简单指令和复杂指令的区分通过 **Prompt 设计**实现：
  - 简单指令：用简短的 Prompt，限制输出格式
  - 复杂指令：用详细的 Prompt，要求逐步推理
- 这样既能节省 Token，又不需要切换模型

#### 9. RAG检索的"检索词"来源
**答复**：**需要建立标准化人群标签库**。
- 在 Python AI 模块维护一个标签映射文件：`./ai_config/tags.json`
- 示例：
```json
{
  "25-35岁女性": "female_25_35",
  "18-25岁男性": "male_18_25",
  "一线城市": "city_tier1"
}
```
- 解析指令时，先将自然语言映射为标准标签，再用标签去检索案例

#### 10. 用户画像数据的初始来源

**答复**：第一阶段**不关注执行调度**。

- **第三阶段**再实现定时调度（Quartz/XXL-JOB）

#### 12. 高风险策略的2小时超时机制
**答复**：第一阶段**不实现超时逻辑**。
- 高风险策略标记为「待人工确认」后，一直等待人工操作
- **第四阶段**再引入延迟队列（Redis ZSet/RabbitMQ 延迟消息）实现超时失效

### 3. 用户登录的密码加密方式
**答复**：**同意使用 BCrypt 加密**。
- 即使第一阶段是演示，也应该用真实加密方式
- Java 后端直接用 Spring Security 的 BCryptPasswordEncoder
- 预设几个测试账号密码（如 admin/123456 的 BCrypt 哈希值）

### 4. 指令ID和策略ID的生成规则

**答复**：**同意每天重置从 0001 开始**。
- 规则：`前缀 + yyyyMMdd + 4位序号`
- 示例：
  - 当天第一条指令：`CMD202603130001`
  - 当天第二条指令：`CMD202603130002`
- Java 后端可以用 Redis 的 INCR 命令按日期生成，或者用数据库自增+日期组合

---

### 5. 前端技术选型的UI库
